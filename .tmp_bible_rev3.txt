 ## Search engine contract (keyword + OpenAI query-expansion semantic + hybrid)

### Hard rules (apply to every search)
- Advanced Search filters are applied as HARD filters first, then ranking is computed.
- Include ONLY experts whose profile visibility state is 'visible'. Hidden/pending experts must never appear in search results (regardless of filters).
- Dependability rating is a BOOST/TIE-BREAKER only (NOT a hard filter).
- When FAQ intent strongly matches a category (via query expansion), category-matched experts receive a mild ranking boost.

### Inputs
- User query text `q` (keywords and/or a natural-language request).
- `mode` ∈ { 'keyword', 'semantic', 'hybrid' }.
- Optional filters from Advanced Search:
  - category_id
  - profession (text)
  - skills (up to 10 tags)
  - min rating
  - max rate
  - quick filters: Verified Experts, Online Now, Available Now
- Pagination/limit:
  - deterministic limit (e.g. top 50) and cursor/offset pagination.

### Search modes
1) `keyword`
- Use keyword/FTS retrieval across the expert “search document” (defined below) using the original user query `q`.

2) `semantic` (implemented as query expansion)
- Use the OpenAI API to expand the user query into a small structured payload, then run keyword/FTS retrieval using the expanded keywords/hints.

3) `hybrid`
- Run BOTH keyword retrieval (original `q`) and semantic retrieval (query-expansion using OpenAI), then merge the ranked lists using RRF.

### Search document (what keyword/FTS MUST search)
Keyword retrieval MUST search across an expert “search document” built from:
- expert_profiles.profession
- categories.name (joined via expert_profiles.category_id)
- expert_profiles.qualifications
- expert_profiles.expert_bio
- expert_profiles.about_services
- expert_profiles.skills_specializations (each tag is independently matchable)
- FAQ entries: include faq_entries.question + faq_entries.answer (for intent discovery and category boosting)

### Semantic query expansion (OpenAI API + caching + timeout)
For every search where mode is 'semantic' or 'hybrid':
1) Call the OpenAI API to produce a SMALL structured JSON expansion payload.
2) OpenAI call constraints:
   - Timeout: hard limit of 10 seconds.
   - Output: MUST be only a small JSON payload (no prose).
   - Output schema:
     {
       "category_hints": string[],        // 0..N category names/keys (optional)
       "keyword_expansions": string[],   // 0..10 keywords/synonyms (optional)
       "skill_hints": string[],          // 0..10 skill/tag hints (optional)
       "confidence": number              // 0..1
     }
3) Caching:
   - Cache expansion results keyed by (normalized_user_query, locale, mode_version).
   - Reuse cached expansions to avoid repeated OpenAI API calls for equivalent queries.
   - Refresh/invalidate cached expansions when FAQ/admin-edited categories or searchable profile fields change.
4) Fallback:
   - If the OpenAI API times out or fails, semantic mode MUST fall back to keyword mode using the original `q`.
5) Use of expansions:
   - Treat keyword_expansions + skill_hints (+ category_hints when applicable) as the inputs for keyword/FTS retrieval and as SOFT boosts (not hard filters) in ranking.

### Candidate retrieval
- Keyword retrieval: retrieve candidates using keyword/FTS over the search document with query `q`.
- Semantic retrieval: retrieve candidates using keyword/FTS over the search document with:
  - the expanded keywords (keyword_expansions)
  - optional additional boosts from category_hints and skill_hints.

### Ranking and merge (deterministic)
- Apply ranking boosts for relevance ordering (e.g., verified experts, online/available/auto-book when used in filters).
- Final ordering must be deterministic:
  - `keyword` mode: order by keyword/FTS relevance + boosts; dependability is a boost/tie-breaker only.
  - `semantic` mode: order by relevance from expanded-keyword retrieval + boosts; dependability is a boost/tie-breaker only.
  - `hybrid` mode: merge keyword-mode and semantic-mode ranked lists using RRF.
- RRF:
  - Use standard RRF parameters (no custom tuning required unless later decided).
  - Dependability remains a boost/tie-breaker only, never a hard filter.
- Results must be ordered deterministically with stable tie-breakers and support pagination (limit + cursor/offset).

Profile visibility state machine (v2):
- visible
- expert_pending_admin_review (optional)
- expert_hidden_incomplete_fields (optional)
- expert_hidden_payment_setup_incomplete (optional)
- learner_hidden_incomplete_fields (optional)
- learner_hidden_email_unverified (optional)
- hidden_unknown_or_error (optional fallback)

Transitions:
- Expert:
  - After expert registration is submitted for admin approval, state becomes `expert_pending_admin_review`.
  - Admin approval => state becomes `visible` (expert becomes publicly searchable/visible).
  - Admin rejection => state remains not-visible; reasons are shown based on what is missing/invalid (see reasons list below).
- Learner:
  - Learner is `visible` only when all required learner fields are complete AND the learner email address is verified.
  - If required learner fields become incomplete or email becomes unverified, state becomes the appropriate learner-hidden state.

Reasons list (what to show in the header for the owner):
- If `expert_pending_admin_review` => “Awaiting admin approval.”
- If `expert_hidden_incomplete_fields` => “Missing required expert registration fields:”
  - Professional Title
  - Category
  - Skills & Specializations (Area of expertise)
  - Professional Bio
  - (If applicable) Required booking inputs needed for approval (e.g., Booking Preferences Rate)
- If `expert_hidden_payment_setup_incomplete` => “Payment/payout setup is incomplete:”
  - Hourly rate / booking rate is missing
  - Stripe payout setup is incomplete (Stripe account/payout connection missing)
- If `learner_hidden_incomplete_fields` => “Missing required learner profile fields:”
  - First name
  - Last name
- If `learner_hidden_email_unverified` => “Verify your email address to continue.”
- If `hidden_unknown_or_error` => “Profile temporarily unavailable. Try again later.”

Owner UX rule:
- Owner sees “Profile temporarily unavailable” plus the reasons list above.
- Non-owners do not see the reasons list; they only see the public gating behavior described in the /expert/:id and /learner/:id sections.

### Expert profile — global promotions (first-session discount & packages)

**Scope:** These are **market-wide** settings (“any eligible learner”), configured on the **Expert Profile** (or equivalent **Booking / pricing** settings). They are **not** per-learner **special booking** rows (`freelance_work`).

**Registration vs profile:** Onboarding (expert registration / setup wizard) may **collect initial values**, but the **live source of truth** is whatever the Bible names for published expert config (e.g. **`expert_profiles`** + **`expert_availability`** and/or **`expert_packages`**). Edits on the profile **immediately** affect what new learners see (subject to validation and moderation rules).

**First-session discount (FSD):** Expert enables FSD with **rules** (e.g. percent or fixed amount; optional **session-length** constraint if product requires). **Eligibility** is enforced **server-side** (e.g. learner has **not** already had a **qualifying completed/paid session** with this expert — define “qualifying” in the Bible). At **checkout**, when FSD applies, write or reserve **`discount_redemptions`** per “discount_redemptions” section; **void** if payment fails or booking is cancelled before charge.

**Packages:** Experts define **sellable packages** as **`expert_packages`** rows (one or many). **Purchase** uses the **package checkout** path only (not special-booking accept). On successful payment: **`transactions`** + **`learner_package_credits`**; redemption uses **`package_credit_redemptions`** and atomic credit decrement as in “Packages — credits and purchase.”

**Learner-facing:** Public **Expert Profile** shows FSD and purchasable packages when enabled and the expert is visible per visibility rules. **Traditional session booking** checkout applies FSD when eligible; **package** CTA opens package purchase flow.

**Implementation alignment:** See **“Implementation contract — single source of truth”** (B1d) and **Table** fields in B1b.

Set up email/SMS notifications with an explicit auth boundary:

- Auth boundary:
  - Use Supabase Auth for user identity.
  - Do not rely on “public” client writes for notification logic.
  - All outbound email/SMS (and template resolution) runs only on the server (Route Handler / server action / trusted backend), using the Supabase **service role** or equivalent server secret — never from the browser.

- Implementation options (v2 — pick one primary pattern; document which in deployment runbooks):
  - **Option A — DB trigger + HTTP:** On `INSERT` into `messages`, a database trigger invokes `notify_new_message()` (or equivalent) which calls the app via `pg_net` / `http_post` (or Supabase Edge Function that forwards to the app).
  - **Option B — DB trigger + Edge Function:** Trigger calls a Supabase Edge Function that sends mail/SMS directly from the function (no intermediate Express requirement).
  - **Option C — Outbox queue:** Trigger inserts into `notification_outbox`; a worker/cron drains the queue. (Use when you need retries and stricter ordering.)

- DB-triggered events (inbox / new message):
  - When a new row is inserted into `messages`, the system must notify the **non-sender** participant of the parent `conversations` row.
  - Webhook / worker payload MUST use **Bible v2 column names**:
    - `message_id` (PK of `messages`)
    - `conversation_id` (FK to `conversations`)
    - `sender_id` (FK to `users.user_id`)
  - The server resolves recipient by loading `conversations.expert_user_id` and `conversations.learner_user_id` and selecting the user where `sender_id` is the other party.
  - Do **not** document camelCase JSON (`messageId`, etc.) as canonical; if the wire format uses camelCase, specify an explicit mapping table in the implementation appendix only.

- Canonical HTTP endpoints (notification + cron) — paths are stable whether the app is Express or Next.js Route Handlers; only the host/base URL changes per environment:
  - `POST /api/notifications/webhook/message` — invoked by DB trigger, Edge Function, or outbox worker. **Body (JSON):** `{ "message_id", "conversation_id", "sender_id" }`. **Auth:** require a shared secret (e.g. header `Authorization: Bearer <NOTIFICATION_WEBHOOK_SECRET>` or `X-Webhook-Secret: <…>`). Reject unauthenticated calls.
  - `GET` or `POST /api/notifications/check-booking-reminders` — invoked by an external **cron** (Render/Railway/Vercel Cron, GitHub Actions, etc.). **Auth:** require a cron secret (query param or header, e.g. `CRON_SECRET`). Reject unauthenticated calls.
  - **Booking reminder selection rule (v2):** query `bookings` using Bible fields `expert_user_id`, `learner_user_id`, `session_date`, `start_time`, and **only** statuses that mean “session is still expected to occur” (define explicitly: e.g. `upcoming` and, if used, `payment_status` / confirmation flags per Bible). Do **not** reference legacy v1 values such as `confirmed` unless the Bible explicitly lists them as aliases.

- Cron-based reminders:
  - Schedule the cron job to call `GET` or `POST /api/notifications/check-booking-reminders` at least every **1–2 minutes** (or a longer interval if the time window in code is widened accordingly).
  - Reminder send time is computed from `session_date` + `start_time` in the user’s display timezone rules per Bible (typically derived from `users.time_zone` / hometown rules).
  - Optional: keep a SQL helper (e.g. `check_booking_reminders()` returning candidate rows) **only** as an implementation detail; the **contract** in the Bible is the HTTP cron endpoint + v2 column names.

  - For any endpoint that sends notifications or marks notification state:

RULES AND DEFINITIONS

#Dependability Rating
A “Dependability Score” should be assigned to each user after every completed (or canceled) booking
A user’s “Dependability Rating” is the average of every “expert_dependabilty” or “learner_dependability” score they receive

The dependability score for each booking is based on a total possible score of 100 points. Points are deducted as follows:
- 100 point deduction if a user doesn’t join a booked session (without cancelling or attempting to reschedule)

- 50 point deduction for cancelling a booking more than 24 hours before scheduled start
- 60 point deduction for cancelling a booking more than 12-24 hours before scheduled start
- 70 point deduction for cancelling a booking more than 6-12 hours before scheduled start
- 80 point deduction for cancelling a booking more than 2-6 hours before scheduled start
- 90 point deduction for cancelling a booking more than 1-2 hours before scheduled start
- 95 point deduction for cancelling a booking more than 0-60 minutes before scheduled start

- 10 point deduction for suggesting a rescheduled time more than 24 hours before scheduled start
- 20 point deduction for suggesting a rescheduled time more than 12-24 hours before scheduled start
- 30 point deduction for suggesting a rescheduled time more than 6-12 hours before scheduled start
- 40 point deduction for suggesting a rescheduled time more than 2-6 hours before scheduled start
- 55 point deduction for suggesting a rescheduled time more than 1-2 hours before scheduled start
- 70 point deduction for suggesting a rescheduled time more than 0-60 minutes before scheduled start

- 5 point deduction joining a call 1-3 minutes late
- 10 point deduction joining a call 3-5 minutes late
- 20 point deduction joining a call 5-10 minutes late
- 50 point deduction joining a call more than 10 minutes late


### Definitions: bookings and special bookings

**Traditional booking** — Session scheduled from the calendar; `bookings` row; standard session pricing & ledger.

**Special booking** — A learner–expert commercial arrangement that is **not** a traditional calendar session checkout. Deliverable-type special bookings (freelance, prep/review, scoped work with a deadline) use **`freelance_work`** and the lifecycle in “Special bookings — lifecycle” below.

**Expert package purchase** — Handled via **`expert_packages`**, **`learner_package_credits`**, **`package_credit_redemptions`**; **not** the same UI lifecycle as deliverable special bookings (no “mark complete by deadline” for the package sale itself). See “Packages — credits and purchase.”

**discount_redemptions** — Discount eligibility tracking only; **not** a special booking.

**booking_offers (optional): on learner accept create `freelance_work` with status “offered” and proceed to payment.


### Special bookings — lifecycle (deliverable / freelance_work)

- Expert sends a **special booking offer** to the learner with **proposed deadline** (and scope/price per Inbox/offer UI).
- Learner **accepts** or **declines**.
  - If **declines:** prompt learner to **message the expert** for a revision; expert may send a **revised offer** (same `freelance_work` row updated or new row with `supersedes_freelance_id` — match schema).
  - If **accepts:** charge learner **total_charge** using the **non-subscription marketplace fee stack** (taxable base **B** = negotiated offer total before platform/tax; platform 10%; taxes 6% on (B + platform fee)). Write **`transactions`** with `transaction_type` for special/freelance charge and **`freelance_id`** set. **Escrow:** funds held until completion acceptance or timeout below.
- After **successful payment:** notify expert; show special booking on expert dashboard under upcoming **Bookings** or **Special bookings** as **In progress**, with **deadline**, **Mark as complete**, link to conversation/`booking_id` if applicable.
- Expert **Mark as complete** (by deadline + grace): optional message and attachments to learner.
- Learner **accepts** or **declines** completion.
  - **Accept:** **release payout** to expert (**expert_earnings** = **B** per ledger).
  - **Decline:** learner must message expert with problem; expert **3 calendar days** to rectify; unresolved → **admin_review**.
- Expert **misses work deadline:** **admin_review**; **3 calendar days grace** after deadline; if still incomplete → **refund learner** (full or partial per policy; until policies exist, **admin case-by-case**).
- Learner **no response** to completion within **3 calendar days** of expert submit → **auto-release payout** to expert (same as accept).

### freelance_work — status enum, transitions, keys, payment

**Statuses:** `offered` | `declined` | `accepted_pending_payment` (omit if accept+charge are atomic) | `paid_in_progress` | `completion_submitted` | `completed` | `refunded` | `admin_review`

**Transitions:**
- `offered` → `declined` | → `accepted_pending_payment` → `paid_in_progress` (skip middle if payment synchronous).
- `declined` → `offered` (re-offer).
- `paid_in_progress` → `completion_submitted` | → `admin_review` (missed deadline path).
- `completion_submitted` → `completed` (learner accepts OR 3-day silence) | → `admin_review` (learner declines) | → `refunded` (if policy allows).
- `admin_review` → `completed` | `refunded`.

**Foreign keys:** Include **`booking_id`** when tied to an existing traditional booking; else nullable with required **`conversation_id`** and/or originating **`message_id`** per product rules. Every successful charge links **`transactions`** to **`freelance_id`**; store **Stripe PaymentIntent id** on `transactions` or `freelance_work` per schema.

**Payment model:** Charge **total_charge** on learner **accept**; **transfer expert payout (B)** on learner **accept completion** or **3-day silence** after expert marks complete; refunds per deadline/dispute/**Admin — disputes and refunds**.


#Packages
- an expert should be able to offer a “package” of sessions, by defining the number of sessions, the duration of each session, and a fixed package price. 
- a package can be offered globally (ie to every learner) or as a special offer to a single learner
- if an expert opts to offer a package, treat it as a credit system. ie, expert can choose a number of sessions and a total price. User will be billed for the total price up front, then given credits to book each session with the expert.
- when a user has credits with a specific expert, who a notice on their dashboard in the overview section and at the top of the “upcoming bookings” dialogue that says “You have [x] more sessions in your package with [Expert’s full name]. Click here to book.” Add a hyperlink to the expert’s profile page for “click here.” Allow users to select and book time (in allotted duration) on the expert’s profile page. On the Payment dialogue, instead of a payment amount, show the text “Session [x] of [your total] sessions in your package.” And allow them to make the booking without making a payment. 
- if an expert offers a global Package, include a badge on the Expert’s profile page that includes the basic parameters of the offer

Decisions:
- Package credits are in-scope for v2 and must be implemented.
- Credits expire 1 year from purchase/grant date.
- First-session discount is one-time per learner per expert.
- If discounted booking is cancelled/refunded, discount eligibility is still consumed.
- Credit-to-rebook amount = full booking total.

- Do not describe **package purchase** as following the **special booking** completion/payout lifecycle unless intentionally extended.
- Do not use a generic **offers** staging table for deliverable special bookings; use **`freelance_work`** from first send (**Option B**).
- Align **Table: transactions** field list with **`transaction_type`**, **`freelance_id`**, **`package_id`**, **`booking_id`** as in marketplace ledger sections.

### expert_packages / learner_package_credits / package_credit_redemptions

**Package purchase:** **Learner** checks out on the **package purchase** flow (separate from traditional session checkout and separate from **special booking** accept). On successful payment: create **`transactions`** (`transaction_type = package_purchase`, `package_id`, amounts per non-subscription fee stack); create **`learner_package_credits`** in the same DB transaction (`remaining_credits`, `expiration_at`).

**Redeeming a credit:** On booking confirmation using a credit, **atomically** decrement **`remaining_credits`** and insert **`package_credit_redemptions`** in the **same transaction** as the booking. Enforce **`bookings`** duration vs **`expert_packages.session_duration_minutes`** unless upgrade rules exist.

**Expiration:** Unused credits **expire** at `expiration_at` (**hard block**). Notify learner at **1 month**, **2 weeks**, **1 week**, and **3 days** before expiration.

**Expert earnings:** On package purchase **`transactions`**, **expert_earnings = B** (package taxable base); platform/tax are in **total_charge**, not deducted from B for payout unless Bible is amended.

### discount_redemptions

- Row written at **checkout** when first-session (or applicable) discount is **applied**.
- **Unique(expert_user_id, learner_user_id)** for that program.
- If payment **fails** or booking **cancelled** before successful charge: **remove** or **void** row — redemption **not consumed**.
- **Promo codes** (future): separate from this table; do not share uniqueness with first-session unless designed.

First Session Discounts
- an expert should be able to offer a first session discount either: 1) globally (ie to every learner) via the expert registration process or by updating preferences from the dashboard, or 2) as a special offer to a specific learner via the “send an offer” dialogue
- the parameters can be set by the expert either as a fixed rate and fixed time, or as a percentage discount for any amount of time
- the discount should be applied to the learner (or learners) for the first completed booking between that learner and the expert
- if the booking is cancelled or rescheduled, the discount should carry over to the first actual session
- if an expert offers a global First Session Discount, include a badge on the Expert’s profile page that includes the basic parameters of the offer


### message_response_times / expert_response_time_stats

**Implementation:** **Database trigger** on `messages` INSERT (preferred), or **trusted server** on each insert — **never** browser-only.

**Rules:** Consecutive learner messages before expert reply → interval starts at **first** learner message in that run. **Exclude** system messages. **Exclude** edited/deleted messages for measurement; no retroactive recompute of old intervals.

**Legacy:** No v1 backfill; stats from v2 go-live.

**UI:** e.g. show average only when `response_interval_count >= 3`, else “New” (per existing Bible copy).


### Flows — global FSD and packages

**E1 — Expert configures (profile / booking settings)**  
1. Expert opens **Expert Profile** or **Booking preferences** (single app surface per B1d).  
2. UI loads **current** FSD fields and **expert_packages** list from **GET** (authenticated, owner).  
3. Expert saves → **one** **PUT/PATCH** (or RPC) persists FSD columns + upserts/deletes **expert_packages** per validation.  
4. Response is **source** for UI; no “success” without persisted read-back.

**E2 — Expert onboarding (registration / setup wizard)**  
Same as **E1** after account creation: call the **same** persist API as profile edit (or a single “complete onboarding” endpoint that internally uses the same service). **No separate** client-only object for packages that never hits the DB.

**L1 — Learner views expert**  
Public **GET** expert profile returns **sanitized** FSD summary (if enabled) and **published** **`expert_packages`** only (no draft rows).

**L2 — Learner books traditional session with FSD**  
1. Learner selects slot and duration → server **quotes** price using **hourly rate × duration**, then applies FSD **only if** expert FSD enabled **and** eligibility query passes.  
2. Client displays quote; learner confirms → create **checkout session** or **PaymentIntent** with **metadata** (`expert_id`, `learner_id`, `booking_id` or draft id, `fsd_applied: true/false`).  
3. On **payment success**: finalize **`bookings`**; insert **`discount_redemptions`** as **`consumed`** (or **`pending` → `consumed`** in one transaction with booking commit).  
4. On **payment failure** or **cancel before capture**: **void** any **`discount_redemptions`** row for that attempt (B3).

**L3 — Learner purchases a package**  
1. From profile, learner selects package → **package checkout** (not session checkout).  
2. On success: **`transactions`** (`package_purchase`) + **`learner_package_credits`** in one DB transaction (B2).  
3. Confirmation email/UI shows **credits** and **expiration**.

**L4 — Learner books using a package credit**  
1. At session booking, learner chooses **pay with credit** (if credits exist for that expert).  
2. On confirm: **atomic** decrement **`remaining_credits`** + insert **`package_credit_redemptions`** + create **`bookings`** (B2). **Do not** apply FSD to the marginal price of a credit session unless the Bible explicitly allows stacking (default: **no stacking**).

**L5 — Expert disables FSD or unpublishes a package**  
Existing **learner_package_credits** and **consumed** **`discount_redemptions`** are **unchanged**; only **new** quotes are affected. Unpublishing does **not** revoke already-purchased credits unless a separate policy is added.

### Implementation contract — registration, profile, and APIs

**Single write path:** FSD fields and **`expert_packages`** may only be changed through **one** server-side code path (e.g. `updateExpertPricingAndPackages(expertId, payload)`), invoked by: expert registration completion, setup wizard finish, availability page save, and profile editor save. **Forbidden:** duplicating save logic in the client that writes different columns or omits packages.

**Single read model:** The app loads promotions for experts via **one** mapper from DB → DTO → UI. Public profile, search cards, and checkout **quote** service all use the **same** field definitions (no parallel `sampleData` / hardcoded package arrays for production).

**Registration = initialize, not fork:** Expert registration may **pre-fill** forms, but on submit it must **persist** through the same API as profile updates. After signup, there is **no** “registration-only” package or FSD state.

**Checkout authority:** **Price, FSD application, and eligibility** are computed **only** on the **server** (or trusted edge function). The client may display a quote but **cannot** send a final discounted amount that the server does not recompute.

**Testing / acceptance:** Add checklist items: (1) create package in wizard → appears in DB and on public profile; (2) edit package on profile → wizard re-open shows same data; (3) FSD quote matches DB config; (4) failed payment does not leave **`discount_redemptions`** consumed; (5) credit booking decrements credits exactly once.


#UserType
To avoid confusion: everyone who registers is considered a user. 
Every user is also a learner, meaning that they are allowed to book experts. 
The term “learner” should only be used to define the “booker” of experts, all “learner” data is stored in the “user” table.
Some users are also Experts. Only users with approved expert profiles can book sessions with learners.
An expert should be able to “Switch to Learning”, ie switch from their Expert Profile to their Learning Profile, if they want to book sessions with an expert.

### Session pricing & ledger (canonical)

**Definitions (currency per deployment, store consistent decimal rules e.g. half-up to cents):**
- **num_15min_blocks** = integer count of 15-minute increments in the booked session (from start_time/end_time on session_date).
- **booking_amount** (booking fee) = expert **rate** (per 15 minutes) × **num_15min_blocks**, minus any applicable **discounts** (first session, package credit, promo), applied in an order defined in product rules.
- **platform_fee** = 10% × **booking_amount**.
- **subtotal_pre_tax** = **booking_amount** + **platform_fee** (learner sees this as one **Subtotal** in checkout; **Cost totals** breaks out the two lines).
- **taxes_fees** = 6% × **subtotal_pre_tax**.
- **total_charge** = **booking_amount** + **platform_fee** + **taxes_fees** (Stripe charge amount).

**Payout:**
- **expert_earnings** for the session transaction = **booking_amount** (the expert's share before platform fee/tax; platform fee and taxes are not part of expert payout unless Bible is later amended).

**Single source of truth:**
- Implement pricing in **one shared server-side function** used by checkout, Stripe PaymentIntent creation, webhooks, and DB writes so UI, `bookings` snapshot fields, and `transactions` never diverge.
- **`transactions`** is the **ledger** for completed charges; **`bookings`** may duplicate key amounts for convenience but **learners' transaction history** reads from **`transactions`**.

**Stripe:**
- PaymentIntent amount = **total_charge**; metadata should include booking_id and breakdown fields for support and reconciliation.


Rules:
- One primary **charge** row per successful session payment (unique on booking_id unless Bible defines partial captures/refunds as separate rows).
- Row is created or finalized when payment succeeds (e.g. Stripe webhook), not from the browser alone.
- **booking_amount**, **platform_fee**, **taxes_fees**, **total_charge**, **expert_earnings** must match the canonical **Session pricing & ledger** formulas.
- **extensions_amount** (if used) follows the same fee/tax rules on extension increments or is documented in extensions section; sum with base session per Bible.
- Refunds/disputes: update **status** and/or append adjustment rows per admin/Stripe policy (specify in Stripe/refund subsection when added).

### Non‑subscription charges (extensions, freelance, packages, offers)

**Scope:** Any one‑time or non‑recurring learner charge processed as a Stripe PaymentIntent (or equivalent) **except** subscription plans. Each charge produces a **`transactions`** row (or adjustment row) so learner history, expert payout, and tax reporting stay consistent.

**Charge categories (examples — use `transaction_type` or parallel FKs in schema):**
- **session_booking** — primary session; `booking_id` required; formulas in **Session pricing & ledger**.
- **session_extension** — additional 15‑min blocks after initial booking; `booking_id` required. **Taxable base** = expert rate × extension blocks (after any extension discount rules). Apply **same** platform % and tax % as session booking. Store extension portion in **`transactions.extensions_amount`** (or a dedicated column) and/or link to a child line; expert **earnings** for that row = extension **booking_amount** (taxable base before platform/tax), unless Bible defines a bundle rule.
- **freelance_work** — `freelance_id` required; **taxable base** = negotiated **total_price** (or `rate` × agreed units per `freelance_work` rules). **Platform fee** = 10% × taxable base; **taxes_fees** = 6% × (taxable base + platform fee); **total_charge** = sum of those three. **expert_earnings** = taxable base (amount released to expert when work is approved), unless a different split is defined.
- **package_purchase** — `package_id` / credit grant; **taxable base** = package **total_price** from **`expert_packages`** (after discounts). Same 10% / 6% stack. **expert_earnings** = taxable base (or prorated per Bible when credits are redeemed).
- **custom_offer** / **prep_review** (if separate from `freelance_work`) — same stack: define **taxable base** from the offer row, then platform 10%, tax 6% on (base + platform), **total_charge**, **expert_earnings** = base unless specified otherwise.

**Universal fee stack (default for all categories above):**
- Let **B** = **taxable base** (the amount the expert is “paid for” before platform and tax — session fee, extension fee, freelance total, package price, etc.).
- **platform_fee** = 10% × **B**
- **subtotal_pre_tax** = **B** + **platform_fee**
- **taxes_fees** = 6% × **subtotal_pre_tax**
- **total_charge** = **B** + **platform_fee** + **taxes_fees**
- **expert_earnings** = **B** (for standard marketplace sessions, extensions, freelance, and packages unless Bible states an exception).

**UI:** For every charge type, learner sees **Subtotal** = **B** + **platform_fee** and a **Cost totals** breakdown; then taxes; then **Total** — same pattern as session booking.

**`transactions` table (Bible):** Add **`transaction_type`** (enum) and nullable FKs **`freelance_id`**, **`package_id`**, **`booking_id`** (booking_id populated for session + extension charges). Exactly one “primary” link should be set per row. **Subscriptions** do not use this row type.

**Implementation:** One shared server function **`computeMarketplaceCharge({ taxableBase })`** returns `{ platform_fee, taxes_fees, total_charge }`; category‑specific code supplies **B** only. Do not duplicate percentage logic per product surface.
